# 可观测性

Kirin Gateway 的可观测性由两部分组成：结构化 JSON 日志（tracing）和 Prometheus 指标。

---

## 1. 结构化 JSON 日志

基于 `tracing` + `tracing-subscriber` 实现，所有日志以 JSON 格式输出，方便日志采集系统（ELK、Loki 等）解析和检索。

### 初始化

在 `main.rs` 中配置：

```rust
tracing_subscriber::fmt()
    .json()
    .with_target(false)
    .with_timer(tracing_subscriber::fmt::time::UtcTime::rfc_3339())
    .init();
```

### 日志输出格式

每条日志为单行 JSON，包含以下字段：

| 字段 | 说明 | 示例 |
|---|---|---|
| `timestamp` | UTC 时间（RFC 3339） | `"2026-05-02T08:30:00.123456Z"` |
| `level` | 日志级别 | `"INFO"`, `"WARN"` |
| `message` | 日志消息 | `"路由匹配成功"` |
| `method` | HTTP 方法 | `"GET"` |
| `path` | 请求路径 | `"/api/users"` |
| `upstream` | 目标上游名称 | `"user-service"` |
| `elapsed_ms` | 请求处理耗时（毫秒） | `"12"` |
| `client_ip` | 客户端 IP | `"192.168.1.100"` |

### 关键日志埋点

| 阶段 | 位置 | 级别 | 说明 |
|---|---|---|---|
| 路由匹配成功 | `proxy.rs` `upstream_peer` | INFO | 记录 method、path、upstream |
| 请求处理完成 | `proxy.rs` `logging` | INFO | 记录耗时、上游、客户端 IP |
| 状态锁异常恢复 | `proxy.rs` 各处 | WARN | RwLock poisoned 时自动恢复 |
| Filter 拒绝 | `filter.rs` | INFO | 记录被哪个 Filter 拒绝 |

---

## 2. Prometheus 指标

基于 `prometheus` crate 实现，在网关内部嵌入 HTTP 端点暴露指标，供 Prometheus Server 定时抓取。

### 指标端点

- **路径：** `/metrics`
- **协议：** Prometheus exposition format (`text/plain; version=0.0.4; charset=utf-8`)
- **处理位置：** `proxy.rs` `request_filter` 中短路返回，不经过上游

### 采集的指标

| 指标名 | 类型 | 标签 | 说明 |
|---|---|---|---|
| `kirin_requests_total` | Counter | `method`, `upstream`, `status_code` | 请求总数 |
| `kirin_request_duration_seconds` | Histogram | `method`, `upstream` | 请求延迟分布 |
| `kirin_upstream_errors_total` | Counter | `method`, `upstream` | 上游连接错误数 |
| `kirin_filter_rejects_total` | Counter | `filter_name`, `status_code` | Filter 拒绝数 |
| `kirin_canary_requests_total` | Counter | `route_id`, `method`, `status_code` | 灰度路由请求总数 |
| `kirin_canary_request_duration_seconds` | Histogram | `route_id`, `method` | 灰度路由请求延迟分布 |

### Histogram 分桶

`kirin_request_duration_seconds` 的分桶边界（秒）：

```
0.001, 0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1.0
```

### 采集时机

| 指标 | 采集位置 | 触发条件 |
|---|---|---|
| `requests_total` | `logging` | 每个请求完成时 |
| `request_duration_seconds` | `logging` | 每个请求完成时（记录耗时） |
| `upstream_errors_total` | `logging` | 请求存在错误时 |
| `filter_rejects_total` | Filter 执行链 | Filter 返回 Stop 时 |
| `canary_requests_total` | `logging` | 存在 canary_decision 时（灰度路由请求） |
| `canary_request_duration_seconds` | `logging` | 存在 canary_decision 时（灰度路由请求） |

### 初始化流程

1. `main.rs` 调用 `crate::observability::metrics::init()`，将所有指标注册到全局 `REGISTRY`
2. 请求到达 `/metrics` 时，调用 `collect()` 从 `REGISTRY` 收集并编码为 Prometheus text format

---

## 3. 架构示意

```
请求进入
  │
  ├─ /metrics ──→ collect() ──→ Prometheus text format ──→ 直接返回
  │
  └─ 其他路径
       │
       ├─ upstream_peer  ──→ INFO: 路由匹配成功
       ├─ request_filter ──→ Filter 执行 ──→ filter_rejects_total++
       ├─ upstream 转发
       ├─ response_filter ──→ Cookie Sticky 注入 Set-Cookie
       └─ logging
            ├─ requests_total++
            ├─ request_duration_seconds.observe()
            ├─ upstream_errors_total++ (如有错误)
            ├─ canary_requests_total++ (灰度路由)
            └─ canary_request_duration_seconds.observe() (灰度路由)
```

# 4. 其他可观测项

## 4.1 灰度指标统计

灰度发布场景下，需要统计灰度版本和稳定版本之间的流量差异、延迟对比和错误率，以评估灰度发布的健康状态。

### 灰度指标列表

| 指标名 | 类型 | 标签 | 说明 |
|--------|------|------|------|
| `kirin_canary_requests_total` | Counter | `route_id`, `method`, `status_code` | 灰度路由请求总数 |
| `kirin_canary_request_duration_seconds` | Histogram | `route_id`, `method` | 灰度路由请求延迟分布（分桶与 `kirin_request_duration_seconds` 相同） |

### 采集条件

灰度指标仅在请求存在 `canary_decision`（即该路由配置了灰度发布）时采集。采集位置在 `proxy.rs` 的 `logging` 阶段。

### PromQL 查询示例

```promql
# 灰度流量分布：按 route_id 统计每秒请求数
sum by (route_id) (rate(kirin_canary_requests_total[5m]))

# 灰度 vs 稳定延迟对比（P99）
histogram_quantile(0.99,
  sum by (route_id, le) (rate(kirin_canary_request_duration_seconds_bucket[5m]))
)

# 灰度路由错误率
sum by (route_id) (rate(kirin_canary_requests_total{status_code=~"5.."}[5m]))
/
sum by (route_id) (rate(kirin_canary_requests_total[5m]))
```
