# 路由匹配

## 概述

路由匹配是 Kirin Gateway 数据面的核心能力，负责根据请求路径将流量分发到对应的上游服务集群。整个路由系统由两个协作组件构成：

| 组件 | 源码位置 | 职责 |
|------|---------|------|
| `Router` | `src/data_plane/router.rs` | 路径 → 上游集群的匹配映射 |
| `RouteRegistry` | `src/data_plane/router/router_white_list.rs` | 接口白名单校验，未注册的接口直接拒绝访问 |

两者的协作关系体现在请求处理链路中（`src/data_plane/proxy.rs`）：

```
客户端请求 → KirinProxy.upstream_peer()
                │
                ├─ ① 路由匹配（Router）           → 无匹配 → 502 No route matched
                ├─ ② 白名单校验（RouteRegistry）→ 未注册 → 403 Forbidden
                └─ ③ 查找上游集群 + 负载均衡选节点 → 无节点 → 502 Bad Gateway

注：HTTP 方法校验在 Filter Chain 的 MethodFilter 中执行（request_filter 阶段）
```

## 匹配策略

路由匹配支持三种策略，通过配置文件中的 `match_type` 字段指定（默认 `exact`）。

### 精确匹配（Exact）

路径必须完全一致才匹配成功。适用于单资源接口。

```
配置：path: "/api/users"，match_type: "exact"

请求 /api/users     → 匹配
请求 /api/users/123 → 不匹配
请求 /api/users?    → 不匹配
```

### 前缀匹配（Prefix）

请求路径以配置的前缀开头即匹配成功。当多个前缀都能匹配时，**最长前缀优先**。适用于 RESTful 集合接口。

```
配置：path_prefix: "/api/orders"，match_type: "prefix"

请求 /api/orders       → 匹配
请求 /api/orders/123   → 匹配
请求 /api/orders/abc/1 → 匹配
请求 /api/order        → 不匹配（前缀不完整）
```

### 正则匹配（Regex）

使用 Rust `regex` crate 进行正则表达式匹配，支持捕获组提取。多个正则规则同时匹配时，**先声明优先**。适用于复杂路径模式。

```
配置：path: "^/api/users/(\d+)$"，match_type: "regex"

请求 /api/users/123   → 匹配，captures = ["123"]
请求 /api/users/abc   → 不匹配
请求 /api/users/123/orders → 不匹配（$ 锚定结尾）
```

### 匹配优先级

三种策略之间存在明确的优先级关系：

```
精确匹配（Exact）> 正则匹配（Regex，按声明顺序）> 前缀匹配（Prefix，最长前缀优先）
```

配置示例（三条规则同时存在时）：

```yaml
routes:
  - route_id: exact-user
    path: /api/users
    match_type: exact
    upstream: user-service

  - route_id: regex-any
    path: "^/api/\\w+$"
    match_type: regex
    upstream: regex-service

  - route_id: prefix-api
    path_prefix: /api/
    match_type: prefix
    upstream: default-service
```

| 请求路径 | 命中规则 | 原因 |
|---------|---------|------|
| `/api/users` | `exact-user` | 精确匹配优先级最高 |
| `/api/orders` | `regex-any` | 精确未命中，正则次之 |
| `/api/orders/123` | `prefix-api` | 精确和正则都未命中，前缀兜底 |

## 数据结构

### RouteRule — 单条路由规则

定义在 `src/data_plane/router.rs:74`。

```rust
pub struct RouteRule {
    pub route_id: String,          // 接口唯一标识
    pub match_type: MatchType,     // 匹配类型：Exact / Prefix / Regex
    pub path: String,              // 路径模式（exact 为精确路径，regex 为正则表达式）
    pub prefix: Option<String>,    // 前缀（仅 Prefix 类型使用）
    pub methods: Vec<String>,      // 允许的 HTTP 方法，空数组表示放行所有方法
    pub upstream: String,          // 上游集群名称
}
```

### MatchType — 匹配类型枚举

```rust
pub enum MatchType {
    Exact,   // 精确匹配
    Prefix,  // 前缀匹配
    Regex,   // 正则匹配
}
```

### RouteMatch — 路由匹配结果

路由匹配的返回类型，携带上游信息和正则捕获组。

```rust
pub struct RouteMatch {
    pub upstream: String,                // 上游集群名称
    pub route_id: String,                // 匹配到的路由 ID
    pub captures: Option<Vec<String>>,   // 正则捕获组（仅 Regex 类型有值）
}
```

### Router — 路由表

定义在 `src/data_plane/router.rs:88`。路由表持有三类匹配规则，各使用不同的数据结构以优化查询效率。

```rust
pub struct Router {
    exact_routes: HashMap<String, RouteRule>,              // 精确匹配：路径 → 路由规则
    regex_routes: Vec<(Regex, RouteRule)>,                 // 正则匹配：编译后的正则 + 路由规则
    prefix_routes: Mutex<Vec<(String, RouteRule)>>,        // 前缀匹配：按前缀长度降序排列（延迟排序）
    prefix_dirty: AtomicBool,                              // 前缀路由延迟排序标志
}
```

| 匹配方式 | 数据结构 | 查找复杂度 | 设计理由 |
|---------|---------|-----------|---------|
| 精确匹配 | `HashMap<String, RouteRule>` | O(1) | 哈希查找，路径完全匹配时效率最高 |
| 正则匹配 | `Vec<(Regex, RouteRule)>` | O(n) | 正则表达式无法建立高效索引，按声明顺序遍历保证可预测的优先级 |
| 前缀匹配 | `Mutex<Vec<(String, RouteRule)>>` | O(n) | `Mutex` 提供内部可变性以支持延迟排序，按长度降序排列保证最长前缀优先 |

> **延迟排序**：`prefix_routes` 使用 `Mutex` 包装，配合 `prefix_dirty: AtomicBool` 实现线程安全的延迟排序。`add_route` 时仅标记 `dirty`，排序推迟到 `match_route` 首次查询时执行。初始化 n 条前缀路由的排序总代价从 O(n² log n) 降为 O(n log n)（只排一次）。

### RouteRegistry — 接口注册表（白名单）

定义在 `src/data_plane/router/router_white_list.rs:31`。

```rust
pub struct RouteRegistry {
    registered_routes: HashMap<String, RouteEntry>,   // key: route_id，value: 接口注册条目
}
```

```rust
pub struct RouteEntry {
    pub route_id: String,       // 接口唯一标识
    pub path: String,           // 路径模式
    pub prefix: Option<String>, // 前缀（仅 Prefix 类型）
    pub match_type: MatchType,  // 匹配类型
    pub methods: Vec<String>,   // 允许的 HTTP 方法
    pub upstream: String,       // 上游集群名称
    pub applicant: String,      // 申请人
    pub applied_at: String,     // 申请时间（ISO 8601）
    pub description: String,    // 接口场景说明
    pub is_auth: bool,          // 是否需要 JWT 认证
}
```

`RouteRegistry` 内部维护了独立的匹配逻辑（精确 → 正则 → 前缀），用于白名单校验阶段。它的匹配逻辑与 `Router` 类似，但职责不同：`RouteRegistry` 只回答"这个路径是否被注册"，而 `Router` 回答"这个路径应该转发到哪个上游"。

## 核心方法

### Router::add_route — 添加路由规则

```rust
pub fn add_route(&mut self, rule: RouteRule) -> Result<(), RouterErrorBuilder>
```

执行流程：

1. **route_id 唯一性校验** — 遍历三个匹配表中已有的 `route_id`，发现重复时返回 `DuplicateRouteId` 错误
2. **按 match_type 分发存储**：
   - `Exact` → 直接插入 `HashMap`
   - `Regex` → 编译正则表达式，失败时返回 `InvalidRegex` 错误，成功后追加到 `Vec`
   - `Prefix` → 追加到 `Vec`，标记 `prefix_dirty`，排序延迟到首次 `match_route` 时执行

### Router::match_route — 执行路由匹配

```rust
pub fn match_route(&self, path: &str) -> Option<RouteMatch>
```

按优先级依次尝试三个阶段：

```
请求路径 "/api/v2/users/123"
    │
    ├─ 1. 精确匹配：HashMap.get("/api/v2/users/123") → None
    │
    ├─ 2. 正则匹配：按声明顺序遍历
    │     "^/api/v2/(\w+)/(\d+)$" → captures(["users", "123"]) ✓
    │
    └─ 3. 前缀匹配：
         ├─ 检查 prefix_dirty 标志，若为 true 则按长度降序排序（仅首次）
         └─ 遍历有序前缀列表，最长前缀优先命中 → 返回 None
```

### RouteRegistry::resolve_path — 白名单校验

```rust
pub fn resolve_path(&self, path: &str) -> Option<String>  // 返回 route_id
```

匹配顺序与 `Router` 一致（精确 → 正则 → 前缀），但仅返回 `route_id` 用于权限校验。

### Router::routes_summary — 路由摘要查询

```rust
pub fn routes_summary(&self) -> Vec<RouteDTO>  // 用于 Admin API
```

聚合三类匹配表中的所有规则，转换为 DTO 供管理接口返回。

## 错误处理

路由系统定义了 `RouterErrorBuilder` 枚举处理构建阶段错误：

| 错误类型 | 触发条件 | 示例 |
|---------|---------|------|
| `InvalidRegex` | 正则表达式编译失败 | `"^/api/[invalid("` |
| `DuplicateRouteId` | 两条规则使用了相同的 `route_id` | exact 和 prefix 规则都叫 `"dup"` |
| `InvalidMatchType` | `match_type` 值不在 `exact/prefix/regex` 中 | `"wildcard"` |

`GatewayState::from_config` 在构建阶段会提前捕获这些错误，转换为 `StateError` 返回，避免运行时才暴露配置问题。

## 配置化路由

路由规则通过 `config.yaml` 的 `routes` 节点声明，由 `GatewayState::from_config` 在启动时加载并构建 `Router` 和 `RouteRegistry`。

### 完整配置示例

```yaml
routes:
  # 精确匹配
  - route_id: user-profile
    path: /api/users/profile
    match_type: exact        # 可省略，默认值
    methods: ["GET"]
    upstream: user-service
    applicant: zhangsan
    applied_at: "2026-04-20T10:00:00+08:00"
    description: 获取用户资料

  # 正则匹配（捕获用户 ID）
  - route_id: user-by-id
    path: "^/api/users/(\\d+)$"
    match_type: regex
    methods: ["GET"]
    upstream: user-service
    applicant: lisi
    applied_at: "2026-04-20T10:05:00+08:00"
    description: 根据 ID 查询用户

  # 前缀匹配
  - route_id: order-api
    path_prefix: /api/orders
    match_type: prefix
    methods: []
    upstream: order-service
    applicant: wangwu
    applied_at: "2026-04-20T10:10:00+08:00"
    description: 订单相关接口
```

### 配置字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `route_id` | string | 是 | 接口唯一标识，全局不可重复 |
| `path` | string | 条件必填 | 精确路径（exact）或正则表达式（regex） |
| `path_prefix` | string | 条件必填 | 前缀路径（prefix） |
| `match_type` | string | 否 | 匹配类型，可选 `exact`/`prefix`/`regex`，默认 `exact` |
| `methods` | string[] | 否 | 允许的 HTTP 方法列表，为空则放行所有方法 |
| `upstream` | string | 是 | 上游集群名称，必须在 `upstreams` 节点中定义 |
| `applicant` | string | 是 | 接口申请人 |
| `applied_at` | string | 是 | 申请时间，ISO 8601 格式 |
| `description` | string | 是 | 接口场景说明 |

> **注意**：`path` 和 `path_prefix` 的使用取决于 `match_type`。Exact/Regex 使用 `path`，Prefix 使用 `path_prefix`。配置不匹配时会在启动时报错。

### 启动校验

`GatewayState::from_config` 在构建路由表时执行以下校验，任何一项失败都会阻止网关启动：

1. **上游引用校验** — 每条路由的 `upstream` 必须在 `upstreams` 节点中存在
2. **匹配类型校验** — `match_type` 必须是 `exact`/`prefix`/`regex` 之一
3. **路径配置校验** — Exact/Regex 必须提供 `path`，Prefix 必须提供 `path_prefix`
4. **route_id 唯一性** — 不允许重复的 `route_id`
5. **正则合法性** — Regex 类型的 `path` 必须能编译为合法正则表达式

## 集成方式

路由组件在网关中的位置关系：

```
config.yaml
    │
    ▼
GatewayState::from_config()  ← src/control_plane/gateway_state.rs
    │
    ├─ 构建 Router（路由匹配表）
    ├─ 构建 RouteRegistry（接口白名单）
    ├─ 构建 HashMap<String, UpstreamCluster>（集群注册表）
    └─ 构建 RateLimiter（限流器）
    │
    ▼
Arc<RwLock<GatewayState>>    ← 控制面写锁更新，数据面读锁查询
    │
    ▼
KirinProxy                  ← src/data_plane/proxy.rs
    └─ upstream_peer()
        ├─ state.read() 获取共享状态
        ├─ router.match_route(path) → 路由匹配
        ├─ registry.resolve_path(path) → 白名单校验
        └─ state.get_cluster(upstream) → 获取集群
        └─ cluster.select_peer() → 负载均衡选节点
```

## 局限性与演进方向

| 局限 | 当前状态 | 演进方向 |
|------|---------|---------|
| 前缀匹配性能 | O(n) 遍历，延迟排序避免重复开销 | 引入前缀树（Trie），优化至 O(k) |
| 路径参数提取 | 仅正则匹配支持捕获组 | 支持 `/api/users/:id` 风格的路径参数声明 |
| 路由动态更新 | 启动时加载，运行时不可变 | 通过 Admin API 实现运行时增删改路由 |
| 白名单匹配性能 | RouteRegistry.resolve_path 每次都遍历全量 | 与 Router 共享匹配索引，避免重复计算 |
| 方法匹配 | 在 Filter Chain 的 MethodFilter 中校验 | 方法匹配下沉到 Router.match_route，支持同路径不同方法路由到不同上游 |
| 运行时动态更新 | 已支持增量 Diff（通过 diff_update） | 通过 Admin API 实现运行时增删改路由 |
