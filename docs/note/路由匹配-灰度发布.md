# 灰度分流流程

```
match_route(path, canary_ctx)
    │
    ├─ 阶段 1：路径匹配 → 收集所有路径命中的路由（collect_path_matches）
    │     精确匹配查找 → 正则匹配遍历 → 前缀匹配遍历
    │
    ├─ 单条命中？→ 直接返回（无灰度决策）
    │
    └─ 多条命中 → 阶段 2：灰度分流
          │
          ├─ 2a. 条件匹配优先
          │     遍历候选路由，寻找 match_rules 非空且条件匹配的路由
          │     条件类型：Header（请求头）/ Cookie / Query（查询参数）
          │     规则关系：any（任一规则匹配即命中）
          │
          ├─ 2b. Cookie Sticky 检查
          │     检查请求是否携带 Sticky Cookie
          │     命中时直接返回对应路由（跳过后续权重分流）
          │
          ├─ 2c. 分流值计算
          │     优先使用客户端 IP Hash（确定性分流，同一 IP 始终命中同一版本）
          │     无客户端 IP 时退化到 SystemTime
          │
          ├─ 2d. 权重分流
          │     基于累积权重区间分配流量（使用 2c 计算的分流值）
          │
          └─ 2e. 兜底
                返回无 canary 配置的路由（即 stable 版本）
```

## 五级分流优先级

| 优先级 | 策略 | 说明 |
|--------|------|------|
| 1（最高） | 条件匹配 | 请求携带指定 Header/Cookie/Query 参数时精准命中灰度版本 |
| 2 | Cookie Sticky | 携带 Sticky Cookie 时直接返回对应版本（跳过权重分流） |
| 3 | 分流值计算 | IP Hash 确定性分流（同一 IP 始终命中同一版本），无 IP 时退化到 SystemTime |
| 4 | 权重分流 | 按累积权重区间概率分配流量（使用优先级 3 计算的分流值） |
| 5（兜底） | 无 canary 路由 | 所有分流策略未命中时，返回 stable 路由 |

## 条件匹配规则

灰度条件匹配支持从请求的 Header、Cookie 和 Query Parameter 中提取匹配值：

```yaml
# Header 匹配：携带 X-Canary: true 请求头时走灰度版本
match_rules:
  - type: header
    key: X-Canary
    value: "true"

# Cookie 匹配：携带 env=canary Cookie 时走灰度版本
match_rules:
  - type: cookie
    key: env
    value: canary

# Query 匹配：URL 携带 ?canary=true 参数时走灰度版本
match_rules:
  - type: query
    key: canary
    value: "true"
```

条件匹配采用 **any** 语义：多条规则中任一匹配即命中。

## 分流值计算与权重分流

分流值决定请求落入哪个权重区间。计算逻辑（`compute_route_value`）：

1. **优先 IP Hash**：使用客户端 IP + `DefaultHasher` 计算哈希值，取模 100。同一 IP 始终命中同一版本，保证确定性分流
2. **退化 SystemTime**：无法获取客户端 IP 时，使用 `SystemTime` 纳秒高位（右移 7 位后取模 100），避免 Windows 平台纳秒分辨率约 100ns 导致最低两位恒为 0

权重值为 0-100 的整数，表示该路由获得流量的百分比。多条路由的权重按声明顺序累积计算：

```
路由 A：weight = 80  → 区间 [0, 80)
路由 B：weight = 20  → 区间 [80, 100)
分流值 65 → 命中路由 A（stable）
分流值 85 → 命中路由 B（canary）
```

## 灰度配置示例

```yaml
routes:
  # 稳定版本（兜底）
  - route_id: user-stable
    path: /api/users
    match_type: exact
    upstream: user-service-v1
    canary:
      weight: 80
      match_rules: []

  # 灰度版本（条件 + 权重 + Cookie Sticky）
  - route_id: user-canary
    path: /api/users
    match_type: exact
    upstream: user-service-v2
    canary:
      weight: 20
      match_rules:
        - type: header
          key: X-Canary
          value: "true"
        - type: query
          key: canary
          value: "true"
      stick: cookie
      sticky_cookie:
        name: kirin_canary
        path: /
        max_age: 3600
```

流量走向：

| 请求特征 | 分流结果 | 触发策略 |
|---------|---------|---------|
| 携带 `X-Canary: true` | user-canary | 条件匹配（优先级 1） |
| 携带 `?canary=true` | user-canary | 条件匹配（优先级 1） |
| 携带 Sticky Cookie `kirin_canary=user-canary` | user-canary | Cookie Sticky（优先级 2） |
| 无特殊标记，IP Hash 命中 80% 区间 | user-stable | 权重分流（优先级 4） |
| 无特殊标记，IP Hash 命中 20% 区间 | user-canary | 权重分流（优先级 4） |


# 会话一致性保证

背景：灰度发布场景下，通常需要避免流量漂移，保证用户尽量访问到同一版本，从而保证用户体验一致。

## IP Hash 确定性分流

使用客户端 IP 的 `DefaultHasher` 哈希值作为分流权重输入，保证同一 IP 始终命中同一版本。这是默认的会话一致性机制——不需要额外配置，只要请求携带客户端 IP，分流值就是确定性的。

## Cookie Sticky Session

通过 `stick: cookie` 配置启用。网关自动管理 Cookie 生命周期：

1. **首次访问**：请求不携带 Sticky Cookie，走正常的灰度分流流程（条件匹配 → IP Hash 权重分流 → 兜底）
2. **版本确定后**：`response_filter` 阶段通过 `Set-Cookie` 注入 Sticky Cookie，Cookie 值为命中的 `route_id`
3. **后续请求**：请求携带 Sticky Cookie，`match_route` 在 2b 步骤直接命中对应版本，跳过权重分流

### Cookie 配置

```yaml
canary:
  stick: cookie
  sticky_cookie:
    name: kirin_canary    # Cookie 名称，默认 "kirin_canary"
    path: /               # Cookie Path，默认 "/"
    max_age: 3600         # 过期时间（秒），默认 3600
```

### Set-Cookie 注入示例

```
Set-Cookie: kirin_canary=user-canary; Path=/; Max-Age=3600
```

## Query Parameter 灰度匹配

通过 `type: query` 的 match_rule，从 URL 查询参数中匹配灰度条件：

```yaml
canary:
  match_rules:
    - type: query
      key: canary
      value: "true"
```

请求 `/api/users?canary=true` 时命中灰度版本。Query 匹配与其他条件类型（Header/Cookie）使用相同的 `any` 语义，可以混合配置。

# 灰度指标统计

灰度发布场景下，通常需要统计灰度版本和稳定版本之间的流量差异，以及灰度版本在稳定版本中的运行效果。

### 采集的指标

| 指标名 | 类型 | 标签 | 说明 |
|--------|------|------|------|
| `kirin_canary_requests_total` | Counter | `route_id`, `method`, `status_code` | 灰度路由请求总数 |
| `kirin_canary_request_duration_seconds` | Histogram | `route_id`, `method` | 灰度路由请求延迟分布 |

### 使用场景

- **流量分布监控**：对比 `route_id="user-stable"` 和 `route_id="user-canary"` 的请求速率，判断灰度比例是否符合预期
- **延迟对比**：对比 stable 和 canary 版本的 P99 延迟，评估灰度版本性能
- **错误率监控**：按 `status_code` 维度统计灰度请求的成功率

详细的指标格式和采集架构参见 [可观测性.md](%E5%8F%AF%E8%A7%82%E6%B5%8B%E6%80%A7.md)。
