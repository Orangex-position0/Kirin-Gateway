# 健康检查

## 背景：为什么需要健康检查？
后端服务可能因为各种原因不可用（部署、OOM、网络故障等）。若网关不知道节点已下线，仍然将发送请求，则会导致请求失败。

所以网关要支持健康检查，当后端服务不可用时，网关会自动将节点标记为不可用，并停止向该节点发送请求。

# 架构设计

## Pingora 健康检查机制
`pingora-load-balancing` 内置了健康检查服务，能定期自动剔除故障节点

## 健康检查集成
方案：扩展 Pingora 健康检查机制，通过 Pingora `BackgroundService` 让 `LoadBalancer` 托管健康检查。

## 当前实现状态

### 配置层（`config.rs`）
`UpstreamConfig` 中通过可选字段 `health_check` 控制是否启用：
```yaml
upstreams:
  my-service:
    algorithm: round_robin
    nodes:
      - addr: "127.0.0.1:8081"
        weight: 1
    health_check:          # 可选，不配置则不启用
      interval_secs: 5     # 检查间隔（秒）
      timeout_secs: 3      # 连接超时（秒），默认 3
```

### 健康检查设置（`health_check.rs`）
- `setup_health_check<S>` — 泛型函数，为任意 `LoadBalancer<S>` 配置 TCP 健康检查
  - 创建 `TcpHealthCheck`，设置连接超时
  - 调用 `lb.set_health_check(hc)` 绑定到 LoadBalancer
  - 设置 `lb.health_check_frequency` 控制检查频率
- `create_background_service` — 从 `LoadBalancerKind` 枚举构建 Pingora `BackgroundService`
  - 内部用 `GenBackgroundService::new(name, lb.clone())` 创建后台服务
  - 返回 `Box<dyn ServiceWithDependents>`，通过 `server.add_boxed_service()` 注册

### 集成流程（`main.rs`）
```text
配置加载 → UpstreamCluster::from_config() → build_*_lb()
  ↓ (如果配置了 health_check)
  setup_health_check(&mut lb, hc_cfg)
  ↓
启动时遍历所有集群 → create_background_service() → server.add_boxed_service()
```

### 注意事项
- 健康检查目前仅支持 TCP 连接检查（`TcpHealthCheck`）
- 健康检查在进程内运行，重启后状态丢失
- 未变更的上游集群在增量更新时保持原实例，健康检查状态不会中断
