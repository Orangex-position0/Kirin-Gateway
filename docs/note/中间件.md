# Filter Chain（过滤器链）

## 背景：为什么需要 Filter Chain

在 API 网关中，每个请求/响应都需要经过一系列通用处理，例如：

- **白名单校验**：确认请求路径在接口注册表中
- **HTTP 方法校验**：确认请求方法被路由规则允许
- **JWT 认证**：对需要鉴权的路由进行 RS256 签名验证
- **限流**：基于 IP 的令牌桶限流
- **Header 注入**：为响应添加网关标识头
- **日志记录**：请求和响应日志

如果把这些逻辑全部写死在代理主流程中，会导致：

1. 代理逻辑与横切关注点耦合，难以维护
2. 新增通用处理需要修改代理核心代码，违反开闭原则
3. 无法灵活组合、排序不同处理逻辑

因此，采用 **Filter Chain 模式** 将横切关注点抽象为独立的、可插拔的组件。

> **注意**：项目早期使用 `Middleware` trait + `MiddlewareChain`，后在 v0.1.0 重构为 `Filter` trait + `FilterChain`。旧代码位于 `src/data_plane/middleware.rs`（已废弃），当前实现位于 `src/data_plane/filter.rs`。

## 架构设计

### 核心 Trait 定义

Filter 通过 `Filter` trait 定义统一接口，位于 `src/data_plane/filter.rs`：

```rust
#[async_trait]
pub trait Filter: Send + Sync {
    /// Filter 名称
    fn name(&self) -> FilterName;

    /// 请求阶段过滤器
    async fn request_filter(
        &self,
        ctx: &mut FilterContext,
        request_header: &mut RequestHeader,
        state: &Arc<RwLock<GatewayState>>,
    ) -> FilterResult;  // Continue | Stop(FilterReject)

    /// 响应阶段过滤器（所有 Filter 都会执行，不短路）
    async fn response_filter(
        &self,
        ctx: &mut FilterContext,
        response_header: &mut ResponseHeader,
    );
}
```

**设计要点**：

- `Send + Sync` 约束：确保 Filter 可以安全地跨线程使用（Pingora 是异步多线程架构）
- `async_trait` 宏：将 trait 方法转为异步，支持在 Filter 中执行异步操作
- `FilterResult` 枚举：`Continue` 放行 / `Stop(FilterReject)` 中断，请求阶段可短路
- **共享状态传入**：`request_filter` 接收 `Arc<RwLock<GatewayState>>`，可读取运行时配置

### 两个处理阶段

| 阶段 | 方法 | 短路行为 | 典型用途 |
|------|------|---------|---------|
| 请求阶段 | `request_filter` | 任一 Filter 返回 `Stop` 即中断 | 方法校验、鉴权、限流 |
| 响应阶段 | `response_filter` | **不短路**，所有 Filter 都执行 | 注入响应头、响应日志 |

### 执行模型：正序执行（非洋葱模型）

```
请求阶段（可短路）：
  Method → Auth → RateLimit → Header → Logging
     ↓
  任一返回 Stop → 中断链路，直接返回错误响应

响应阶段（不可短路）：
  Method → Auth → RateLimit → Header → Logging
     ↓
  所有 Filter 都执行完毕
```

## 执行结果类型

### FilterResult — Filter 执行结果

```rust
pub enum FilterResult {
    Continue,             // 继续执行下一个 Filter
    Stop(FilterReject),   // 中断 Filter 链，附带拒绝原因
}
```

### FilterReject — 拒绝原因枚举

```rust
pub enum FilterReject {
    Unauthorized,     // 401 — JWT 认证失败
    Forbidden,        // 403 — 白名单/方法校验失败
    NotFound,         // 404
    MethodNotAllowed, // 405
    TooManyRequests,  // 429 — 限流
    InternalError,    // 500
    Custom { code: u16, reason: String },
}
```

### FilterName — Filter 名称枚举

```rust
pub enum FilterName {
    WhiteList,
    Method,
    Auth,
    RateLimit,
    Header,
    Logging,
    Custom(String),
}
```

## FilterContext — 统一上下文

每次请求创建一个 `FilterContext`，在所有 Filter 之间共享状态：

```rust
pub struct FilterContext {
    pub path: String,                    // 请求路径
    pub method: String,                  // HTTP 方法
    pub client_ip: String,               // 客户端 IP
    pub upstream_name: Option<String>,   // 上游集群名称
    pub route_id: Option<String>,        // 路由 ID
    pub start_time: std::time::Instant,  // 请求起始时间
    pub rate_limit_remaining: Option<usize>,  // 限流剩余令牌
    pub auth_user_id: Option<String>,    // JWT sub claim
}
```

各字段在请求处理链路中的设置时机：

| 字段 | 设置时机 | 设置者 |
|------|---------|--------|
| `path` / `method` / `client_ip` / `start_time` | `request_filter` 阶段初始化 | `KirinProxy::request_filter` |
| `upstream_name` | `upstream_peer` 阶段 | `KirinProxy::upstream_peer` |
| `route_id` | `upstream_peer` 阶段 | `KirinProxy::upstream_peer`（白名单校验后） |
| `rate_limit_remaining` | Filter Chain 请求阶段 | `RateLimitFilter` |
| `auth_user_id` | Filter Chain 请求阶段 | `AuthFilter` |

## FilterChain — 编排器

```rust
pub struct FilterChain {
    filters: Vec<Arc<dyn Filter>>,
}
```

核心方法：

| 方法 | 说明 |
|------|------|
| `add_filter(filter)` | 注册 Filter（按注册顺序执行） |
| `run_request_filters(ctx, header, state)` | 执行请求阶段，任一返回 `Stop` 即中断 |
| `run_response_filters(ctx, header)` | 执行响应阶段，所有 Filter 都执行 |
| `len()` / `is_empty()` | 查询 Filter 数量 |

## 内置 Filter

### 1. WhiteListFilter — 白名单校验

| 属性 | 值 |
|------|---|
| 源码位置 | `src/data_plane/filter/whitelist.rs` |
| 请求阶段 | 检查请求路径是否在 `RouteRegistry` 中注册，未注册则返回 403 |
| 响应阶段 | 无操作 |

> **注意**：当前版本中，WhiteListFilter 虽已实现但**未注册到默认 FilterChain**。白名单校验在 `KirinProxy::upstream_peer` 中直接调用 `state.registry.resolve_path()` 执行。

### 2. MethodFilter — HTTP 方法校验

| 属性 | 值 |
|------|---|
| 源码位置 | `src/data_plane/filter/method.rs` |
| 请求阶段 | 根据 `route_id` 查找路由规则中的 `methods` 列表，方法不在列表中则返回 403 |
| 响应阶段 | 无操作 |

逻辑：
1. 根据 `ctx.route_id` 从 `RouteRegistry` 获取允许的方法列表
2. `methods` 为空数组时放行所有方法
3. 不在列表中时返回 `FilterReject::Forbidden`

### 3. AuthFilter — JWT RS256 认证

| 属性 | 值 |
|------|---|
| 源码位置 | `src/data_plane/filter/auth.rs` |
| 请求阶段 | 对 `is_auth: true` 的路由进行 JWT 认证 |
| 响应阶段 | 无操作 |

请求阶段流程：

```mermaid
flowchart TD
    START[请求到达 AuthFilter] --> CONFIG{auth_config 存在?}
    CONFIG -->|否| PASS[Continue]
    CONFIG -->|是| AUTH{route is_auth?}
    AUTH -->|否| PASS
    AUTH -->|是| TOKEN{提取 Bearer Token}
    TOKEN -->|缺失/格式错误| STOP1[Stop Unauthorized 401]
    TOKEN -->|成功| DECODE[JWT RS256 验证]
    DECODE -->|验证失败| STOP2[Stop Unauthorized 401]
    DECODE -->|成功| FORWARD[透传 Claims 到请求头<br/>X-User-{ClaimName}]
    FORWARD --> PASS

    style PASS fill:#4CAF50,color:#fff
    style STOP1 fill:#F44336,color:#fff
    style STOP2 fill:#F44336,color:#fff
```

JWT Claims 透传机制：
- 配置 `claims_to_forward: ["sub", "exp"]` 时，将对应 claim 以 `X-User-Sub` / `X-User-Exp` 头注入请求
- `sub` claim 同时保存到 `ctx.auth_user_id`

### 4. RateLimitFilter — 令牌桶限流

| 属性 | 值 |
|------|---|
| 源码位置 | `src/data_plane/filter/rate_limit_filter.rs` |
| 请求阶段 | 基于 `client_ip` 执行令牌桶限流检查，超限返回 429 |
| 响应阶段 | 注入 `X-RateLimit-Remaining` 响应头 |

逻辑：
1. 从 `GatewayState` 获取 `RateLimiter`（未配置则直接放行）
2. 调用 `rate_limiter.check(client_ip)` 获取令牌
3. 允许时设置 `ctx.rate_limit_remaining`，拒绝时返回 `FilterReject::TooManyRequests`

### 5. HeaderFilter — Header 注入

| 属性 | 值 |
|------|---|
| 源码位置 | `src/data_plane/filter/header.rs` |
| 请求阶段 | **无操作**（X-Gateway 头在 `upstream_request_filter` 中注入） |
| 响应阶段 | 注入 `X-Powered-By: Kirin Gateway` |

> **重要**：`X-Gateway` 请求头不在 Filter Chain 中注入，而是在 `KirinProxy::upstream_request_filter` 中注入。这是因为 Pingora 文档明确说明：只有 `upstream_request_filter` 中修改的请求头才会传递给上游服务。

### 6. LoggingFilter — 日志记录

| 属性 | 值 |
|------|---|
| 源码位置 | `src/data_plane/filter/logging.rs` |
| 请求阶段 | 记录 `{method} {path}` |
| 响应阶段 | 记录 `Response status: {status_code}` |

## 与代理的集成

### 注册方式

`FilterChain` 在 `GatewayState::build_default_filter_chain()` 中构建：

```rust
fn build_default_filter_chain() -> FilterChain {
    let mut chain = FilterChain::new();
    chain.add_filter(Arc::new(MethodFilter));
    chain.add_filter(Arc::new(AuthFilter));
    chain.add_filter(Arc::new(RateLimitFilter));
    chain.add_filter(Arc::new(HeaderFilter));
    chain.add_filter(Arc::new(LoggingFilter));
    chain
}
```

> **注意**：`WhiteListFilter` 未在默认链中注册，白名单校验在 `upstream_peer` 阶段直接执行。

### 调用位置

| 阶段 | 调用位置 | 方法 |
|------|---------|------|
| 请求阶段 | `KirinProxy::request_filter` | `filter_chain.run_request_filters()` |
| 响应阶段 | `KirinProxy::response_filter` | `filter_chain.run_response_filters()` |

为避免 `RwLockReadGuard` 跨 `.await`，调用前先 clone `FilterChain` 再释放读锁。

## 扩展指南

要新增自定义 Filter：

1. 定义一个结构体
2. 实现 `Filter` trait
3. 在 `GatewayState::build_default_filter_chain()` 中注册

```rust
pub struct CorsFilter;

#[async_trait]
impl Filter for CorsFilter {
    fn name(&self) -> FilterName {
        FilterName::Custom("cors-filter".to_string())
    }

    async fn request_filter(
        &self,
        _ctx: &mut FilterContext,
        _request_header: &mut RequestHeader,
        _state: &Arc<RwLock<GatewayState>>,
    ) -> FilterResult {
        FilterResult::Continue
    }

    async fn response_filter(
        &self,
        _ctx: &mut FilterContext,
        response_header: &mut ResponseHeader,
    ) {
        response_header.insert_header("Access-Control-Allow-Origin", "*").unwrap();
    }
}
```
